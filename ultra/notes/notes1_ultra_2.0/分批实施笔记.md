# Ultra v3.0 全栈升级 — 分批实施笔记

## 总体架构思路

原始项目是纯静态 SPA（HTML/CSS/JS），数据全在 localStorage，题库来自 `data/default.json`。升级目标：Cloudflare 全栈（D1 + KV + Pages Functions），新增用户系统、好友系统、管理面板。

**核心原则**：
- 现有渲染代码不动（`renderChoice`、`renderFillBlank`、`renderBankSelect` 等）
- 未登录用户仍可正常刷题
- localStorage 仍是主存储，API 是增强
- 纯 vanilla JS，不引入框架

**分批策略**：每批可独立验证，每步都有可运行的中间状态。先建后端 API（不影响前端），再逐步接入前端。

**⚠️ 重要变更**：原计划用 R2 存题目，部署时发现 R2 需要绑卡，改为 KV 存储（单值最大 25MB，default.json 才 700KB，完全够用）。第 1 批写的是 R2 方案，实际已全部改为 KV，详见"实际部署记录"章节。

## 第 1 批：基础设施 + API 后端

### 工作流程

1. **先建基础设施**（wrangler.toml、_redirects、schema.sql）— 这些是所有 API 的前提
2. **再建认证中间件**（_middleware.js）— 所有受保护 API 都依赖它
3. **然后按业务域并行建 API**：auth → banks → friends → admin

### 思路细节

#### wrangler.toml
Cloudflare Pages 的配置文件，声明三大绑定：
- `DB`：D1 数据库（存 users、friends、banks 元数据）
- `QUESTIONS`：R2 bucket（存题库 JSON 文件，因为单题库可能几 MB，D1 单行 1MB 限制）
- `SESSIONS`：KV namespace（存 session token → user_id 映射）

关键：`compatibility_date` 和 `compatibility_flags` 确保运行时行为一致。

#### _redirects
单行 `/* /index.html 200`，让 Cloudflare Pages 对所有前端路由返回 index.html（SPA 路由）。但 `/api/*` 路径会被 Pages Functions 优先匹配，不会被重写。

#### schema.sql
三张表的设计：
- **users**：id(UUID)、email(UNIQUE)、nickname、password_hash、role('user'|'admin')
- **friends**：user_id + friend_id 联合主键，status('pending'|'accepted'|'rejected')。好友是双向的，接受时插入两行
- **banks**：id(UUID)、owner_id、name、is_default、is_public、r2_key。r2_key 指向 R2 中的题目 JSON

#### _middleware.js
核心职责：解析请求中的 session cookie，查 KV 拿到 user_id，再查 D1 拿到用户信息，注入 `context.locals.user`。

设计决策：
- 使用 `context.locals` 是 Cloudflare Pages Functions 的约定，后续 handler 可直接取
- 未登录时 `context.locals.user = null`，不阻断请求——让每个 API 自己决定是否需要登录
- Cookie 名 `session_id`，HTTP-only + Secure + SameSite=Strict + 7天过期

#### auth API（register/login/logout/me）
**register.js**：
- PBKDF2-SHA256 哈希密码（Web Crypto API，Cloudflare Workers 原生支持）
- 存储格式 `salt:hash`，salt 是 hex 编码的随机字节
- 注册成功自动创建 session（写 KV + Set-Cookie）
- 特殊逻辑：新用户自动和 admin 加好友（`INSERT OR IGNORE`，幂等）

**login.js**：
- 用相同 salt 对输入密码做 PBKDF2，比对 hash
- 成功后创建 session

**logout.js**：
- 从 KV 删除 session key
- `Set-Cookie` 设 `Max-Age=0` 清除浏览器 cookie

**me.js**：
- 直接从 `context.locals.user` 读，中间件已解析好
- 返回 `{id, email, nickname, role}`

#### banks API（index/[id]/questions）
**index.js**（GET/POST）：
- GET：好友可见性查询是核心难点。SQL 用 UNION 三段：
  1. `is_default = 1`（所有人可见默认题库）
  2. `is_public = 1`（所有可见公开题库）
  3. `owner_id = ?`（自己的题库）
  4. `owner_id IN (好友列表)`（好友的私有题库）
- 未登录用户只能看到前两段
- POST：创建题库元数据记录 + R2 初始化空题目 JSON

**[id].js**（GET/PUT/DELETE）：
- 权限检查：GET 要可见性通过、PUT/DELETE 要是 owner 或 admin
- PUT 可改 name/is_public
- DELETE 删 D1 记录 + R2 对象

**questions.js**（GET/PUT）：
- GET：从 R2 读整个题目 JSON
- PUT：向 R2 写整个题目 JSON（整体覆盖，不做增量）
- 权限：PUT 需要 owner 或 admin

#### friends API（search 除外在 friends/ 目录下）
**search.js**（/api/users/search）：
- 模糊匹配 nickname 或 email，LIMIT 10
- 排除自己

**index.js**（GET /api/friends）：
- 返回已接受的好友列表 + 待处理的请求（别人发给我的 pending）

**request.js**：
- 检查目标用户存在
- 检查是否已是好友 / 已发过请求
- **自动接受逻辑**：如果对方已经给我发了 pending 请求，我此时发请求 → 直接双方 accepted
- 正常情况：插入一条 pending 行

**accept.js**：
- 验证 pending 请求存在（`user_id=对方, friend_id=我`）
- 更新为 accepted + 插入反向 accepted 行

**reject.js**：
- 直接 UPDATE status = 'rejected'，不需要插入反向行

#### admin API
**users.js**（GET/DELETE）：
- GET：LEFT JOIN banks 统计每个用户的题库数
- DELETE：级联删除 R2 对象 → D1 banks → D1 friends → D1 users。不能删自己

**banks.js**（GET）：
- JOIN users 获取 owner_name，管理员视角看所有题库

### 设计决策总结

| 决策 | 选择 | 原因 |
|------|------|------|
| 密码哈希 | PBKDF2-SHA256 | Web Crypto API 原生支持，Workers 环境无 bcrypt |
| 会话存储 | KV + HTTP-only cookie | KV 低延迟读，cookie 防 XSS |
| 题目存储 | R2 | 单题库可能 >1MB，超过 D1 单行限制 |
| 题库元数据 | D1 | 需要关系查询（好友可见性） |
| 好友关系 | 双向双行 | 查询简单，不需要 OR 条件 |
| 自动加好友 | 注册时和 admin 加好友 | 新用户注册后立即可见管理员的题库 |

## 第 2 批：前端 API 层 + 用户认证 UI

### 工作流程

1. **先建 api.js** — 底层依赖，所有前端 API 调用都走它
2. **再建 auth-ui.js** — 认证弹窗逻辑，调用 api.js
3. **然后改 app.js** — 加 state.user 和事件监听，对接 auth-ui
4. **接着改 index.html** — 加 DOM 结构和 script 引用
5. **最后改 style.css** — 追加样式

### 思路细节

#### js/api.js
统一封装所有 HTTP 请求，职责：
- **credentials: 'include'**：确保 cookie 随请求发送
- **JSON 序列化**：body 自动 `JSON.stringify`
- **错误处理**：非 2xx 抛出带 status 和 data 的 Error；网络断开抛出 `offline: true` 的 Error
- **命名空间组织**：`API.auth.*`、`API.banks.*`、`API.users.*`、`API.friends.*`、`API.admin.*`

为什么不直接在各处写 fetch？因为：
- 统一处理 cookie、JSON、错误
- 改一处就改全局（比如以后加 token 刷新）
- 调用方代码更简洁：`await API.auth.login(email, password)`

#### js/auth-ui.js
AuthUI 对象封装弹窗逻辑，职责：
- **open/close/switchTab**：控制弹窗显示和登录/注册 tab 切换
- **handleLogin/handleRegister**：表单校验 + 调 API + 成功后派发 CustomEvent
- **handleLogout**：调 API + 派发 logout 事件
- **showError/clearError**：统一错误提示

关键设计：**用 CustomEvent 通讯**，而不是直接操作 app.js 的 state。
- `auth:login` 事件携带 `detail: user` 对象
- `auth:logout` 事件不携带数据
- app.js 监听事件更新 `state.user` 和 UI

为什么用事件而不是直接函数调用？
- 解耦：auth-ui.js 不需要知道 app.js 的 state 结构
- 可扩展：未来 friends-ui.js 也能监听 `auth:login` 做初始化
- 独立性：auth-ui.js 可以单独测试

#### js/app.js 修改
改动最小化原则，只加了三部分：

1. **`state.user = null`**：在 state 对象中加一个字段
2. **`updateUserUI()`**：根据 `state.user` 是否存在，渲染"登录"按钮或"昵称+退出"
3. **init 改造**：先调 `API.auth.me()` 恢复登录态，再加载题库
4. **事件监听**：`auth:login` 更新 state.user + 调 updateUserUI，`auth:logout` 清除
5. **快捷键屏蔽**：overlays 数组加 `'authOverlay'`

注意：**没有改 `loadBank()` 和 `saveState()`**，这是计划书的要求——第 2 批只做认证，题库仍走 localStorage。

#### index.html 修改
三处改动：
1. **header-right 中加 `<div id="userArea">`**：动态填充登录按钮或用户信息
2. **加 authOverlay 弹窗**：登录/注册表单，tab 切换
3. **加 `<script src="js/api.js">` 和 `<script src="js/auth-ui.js">`**：在 app.js 之前加载（因为 app.js 的 init 会调 API.auth.me()）

script 加载顺序很重要：api.js → auth-ui.js → app.js，因为 app.js 的 init 立即调用了 `API.auth.me()`。

#### css/style.css 修改
追加两类样式：
1. **用户区域**：`.user-name`、`.btn-login`、`.btn-logout`，与现有 header 按钮风格统一（半透明背景 + 白色文字）
2. **认证弹窗**：`.auth-tabs`（底部边框高亮当前 tab）、`.auth-field`（label + input 布局）、`.auth-submit`（蓝色全宽按钮）、`.auth-error`（红色错误提示）

### 设计决策总结

| 决策 | 选择 | 原因 |
|------|------|------|
| 模块通讯 | CustomEvent | 解耦 auth-ui 和 app，可扩展 |
| API 封装 | 全局 API 对象 | 统一 cookie/JSON/错误处理 |
| 登录态恢复 | init 时调 /api/auth/me | 刷新页面不丢登录状态 |
| 离线兼容 | API 失败不阻断 | 未登录/离线仍可刷题 |
| 表单校验 | 前端 + 后端双重 | 前端即时反馈，后端防绕过 |
| script 顺序 | api → auth-ui → app | app init 依赖前两者已加载 |

### 验证清单

- [ ] 打开页面 → header 右侧显示"登录"按钮
- [ ] 刷题功能完全正常（不点登录不影响）
- [ ] 点"登录" → 弹出认证弹窗，默认显示登录 tab
- [ ] 切换到注册 tab → 填写邮箱/昵称/密码 → 注册成功 → 弹窗关闭，header 显示昵称+"退出"
- [ ] 点"退出" → header 恢复"登录"按钮
- [ ] 登录后刷新页面 → 仍显示昵称+"退出"（session cookie 恢复）
- [ ] 密码错误 → 弹窗显示红色错误提示
- [ ] 离线时打开页面 → 显示"登录"按钮，刷题正常（API.me 失败不阻断）

## 第 3 批：好友/管理功能 + API 题库加载

### 工作流程

1. **先建 friends-ui.js** — 好友管理弹窗（搜索用户、待处理请求、好友列表）
2. **再建 admin-ui.js** — 管理员面板（用户表格、题库表格、删除操作）
3. **然后改 app.js** — 这是本批最复杂的修改，三处核心改动：
   - `loadBank()` 加 API 路径（带 localStorage 缓存 + fallback）
   - `saveState()` 加 API 同步（fire-and-forget）
   - `renderBankSelect()` 合并 API 题库
   - 新增 `loadApiBanks()` + 登录/登出事件触发
4. **接着改 index.html** — 加 friendsOverlay/adminOverlay 弹窗 + script 引用
5. **最后改 style.css** — 好友列表/搜索结果/管理表格样式

### 思路细节

#### js/friends-ui.js
FriendsUI 对象，职责：
- **open**：未登录则跳转登录弹窗，否则打开并加载好友列表
- **loadFriends**：调 `API.friends.list()`，分别渲染好友列表和待处理请求
- **search**：oninput 触发，调 `API.users.search()`，实时显示搜索结果
- **sendRequest/accept/reject**：调对应 API，成功后刷新列表

关键 UI 设计：
- 头像用昵称首字母 + 蓝色圆形背景（无需真实头像图）
- 待处理请求用黄色背景（`#fffbe6`）高亮，搜索结果用绿色背景区分
- 自动接受时给特殊提示（`res.auto_accepted`）

#### js/admin-ui.js
AdminUI 对象，职责：
- **open**：非 admin 拒绝，否则加载用户表和题库表
- **loadUsers/loadBanks**：分别渲染两个表格
- **deleteUser/deleteBank**：confirm 后调 API，成功后刷新两个表

关键点：admin 用户的"删除"按钮不显示（不能删管理员），默认题库不允许删除。

#### js/app.js 修改（本批核心）

**1. 新增 `apiBanks` 缓存和 `LS.apiBankCache` key**
- `apiBanks`：登录后从 `/api/banks` 拉取的题库元数据列表
- `LS.apiBankCache(id)`：每个 API 题库的题目缓存，离线时可读

**2. `loadBank()` 加 API 路径** — 三级 fallback：
```
1. localStorage 本地题库（getUserBanks）
2. API 题库（仅登录后）：
   - 先读 localStorage 缓存
   - 缓存无/空则调 API.banks.getQuestions()，写回缓存
3. npoint 缓存的题库（旧逻辑保留）
```
API 题库对象打上 `_api: true` 标记，供 saveState 识别。

**3. `saveState()` 加 API 同步**
```js
if (state.user && state.bank._api) {
  API.banks.putQuestions(...).catch(e => console.warn(...));
}
```
- **fire-and-forget**：不 await，不阻断 UI
- **失败静默**：catch 后只 console.warn，不弹窗
- **仅同步题目内容**：答题记录（wrong/right/fav）仍只存 localStorage

**4. `renderBankSelect()` 合并 API 题库**
- 渲染顺序：默认题库 → 本地题库 → API 题库（去重：已在本地的跳过）
- API 题库的题目数从缓存读取（避免每次渲染都请求）

**5. 新增 `loadApiBanks()`**
- 登录后调用，拉取可见题库列表（过滤掉默认题库，因为前端已硬编码）
- 失败时 `apiBanks = []`，静默降级

**6. 事件监听增强**
- `auth:login`：除了更新 UI，还 `await loadApiBanks()` + `renderBankSelect()`
- `auth:logout`：清空 `apiBanks` + 重渲染侧栏
- init 启动时也调 `loadApiBanks()`

**7. `updateUserUI()` 加好友/管理入口**
- 登录后显示 👥（好友）按钮
- admin 额外显示 ⚙（管理）按钮
- 未登录时不显示这两个按钮

#### index.html 修改
加两个弹窗 + 两个 script：
- **friendsOverlay**：搜索框 + 搜索结果区 + 待处理区 + 好友列表区
- **adminOverlay**：两个 table（用户/题库）
- script 顺序：api → auth-ui → friends-ui → admin-ui → app

#### css/style.css 修改
追加三类样式：
1. **header 按钮**：`.btn-friends`、`.btn-admin`，与 help/import 按钮风格统一
2. **好友弹窗**：`.friend-search`、`.friend-item`、`.friend-avatar`、`.btn-accept/reject/add-friend`
3. **管理面板**：`.admin-table`（表格）、`.admin-badge`（角色标签）、`.btn-admin-del`

### 设计决策总结

| 决策 | 选择 | 原因 |
|------|------|------|
| API 题库缓存 | localStorage（apiBankCache） | 离线可读，减少重复请求 |
| 题目同步策略 | fire-and-forget | 不阻断答题，失败不影响本地 |
| 题库 ID 区分 | String 转换比较 | API 是数字 ID，localStorage 是字符串，统一用 String 比较 |
| 答题记录存储 | 仅 localStorage | 这是个人刷题记录，不需要同步到服务端 |
| API 题库标记 | `_api: true` | saveState 据此决定是否同步，渲染时也可区分 |
| 好友搜索触发 | oninput 实时 | 即时反馈，无需点搜索按钮 |
| 管理入口显示 | role === 'admin' | 前端控制可见性，后端 API 独立鉴权 |
| 离线降级 | API 失败用缓存/默认 | 刷题核心功能不依赖网络 |

### 关键设计：三级题库加载 fallback

这是本批最值得复盘的设计。`loadBank()` 的加载顺序：

```
bankId !== 'default'
  ├─ 1. 本地 localStorage（用户导入的题库）
  │     └─ 找到 → 用
  ├─ 2. API 题库（仅 state.user 存在时）
  │     ├─ 读 localStorage 缓存 → 找到 → 用
  │     └─ 调 API.banks.getQuestions → 成功 → 写缓存 + 用
  │                                     └─ 失败 → 若有缓存用缓存，否则报错
  └─ 3. npoint 缓存（旧逻辑保留兼容）
```

为什么这样设计？
- **本地优先**：用户导入的题库在本地，响应最快
- **API 次之**：服务端题库（含好友的）需要网络
- **缓存兜底**：API 挂了但本地有缓存，仍可刷题
- **npoint 兼容**：保留旧逻辑，不破坏已有功能

- [ ] 未登录 → 侧栏只有默认题库 + 本地题库，无 API 题库

## 第 4 批：数据迁移 + 生产部署

### 工作流程

1. **生成管理员密码哈希** — 运行 gen-admin-hash.js 得到 PBKDF2 哈希
2. **准备 seed.sql** — 将哈希填入占位符
3. **创建 Cloudflare 资源** — D1 + R2 + KV，更新 wrangler.toml
4. **执行建表** — `wrangler d1 execute ... --file=schema.sql`
5. **插入种子数据** — `wrangler d1 execute ... --file=seed.sql`
6. **上传默认题库到 R2** — `wrangler r2 object put`
7. **本地测试** — `wrangler pages dev .`
8. **部署生产** — `wrangler pages deploy`

### 思路细节

#### seed.sql 设计
种子数据只需要两条记录：

1. **管理员用户**：email=admin@example.com, role=admin
   - 密码哈希不能硬编码在 SQL 里，因为 PBKDF2 的 salt 是随机的
   - 所以用 `__ADMIN_HASH_PLACEHOLDER__` 占位，部署时替换
   - 使用 `INSERT OR IGNORE` 保证幂等（重复执行不会报错）

2. **默认题库**：id=1, is_default=1, is_public=1
   - `r2_key = 'banks/default.json'`，指向 R2 中上传的题目 JSON
   - `owner_id = 1`（管理员）
   - 同样 `INSERT OR IGNORE`，幂等

为什么不用 `INSERT OR REPLACE`？因为 `OR REPLACE` 会覆盖已有数据，可能导致意外重置。`OR IGNORE` 只在记录不存在时插入。

#### gen-admin-hash.js
一个小脚本，与 register.js 使用完全相同的 PBKDF2-SHA256 逻辑生成哈希：
- 16 字节随机 salt → base64
- 100000 次 PBKDF2-SHA256 → 256 位 → base64
- 输出格式 `salt:hash`

为什么不在 seed.sql 里直接写一个固定哈希？
- PBKDF2 的 salt 必须随机，否则彩虹表可破
- 每次部署生成不同的 salt，即使密码相同，哈希也不同
- 这与注册流程一致：每个用户的 salt 都是独立的

#### R2 上传
默认题库约 700KB（191 题），直接用 `wrangler r2 object put` 上传即可：
- R2 路径：`banks/default.json`
- Content-Type: `application/json`
- 与 seed.sql 中的 `r2_key` 对应

为什么默认题库不直接放 D1？
- D1 单行最大 1MB，715KB 虽然在限制内，但余量太小
- 题目是纯读场景，R2 更适合存储大 JSON
- 未来题库可能增长，R2 无单文件大小限制

#### wrangler.toml 配置
三个绑定需要实际 ID：
- `database_id`：D1 数据库 ID（`wrangler d1 create` 输出）
- KV `id`：KV namespace ID（`wrangler kv namespace create` 输出）
- R2 只需 `bucket_name`，不需要 ID

#### 本地测试
`wrangler pages dev .` 会启动本地模拟环境，D1/R2/KV 都用本地模拟存储。但注意：
- 本地 D1 是空的，需要先执行建表：`wrangler d1 execute exam-prep-db --local --file=schema.sql`
- 本地 R2 也是空的，默认题库需要本地上传或 fallback 到 `data/default.json`
- Cookie 的 `Secure` 属性在 HTTP 下不生效，本地测试需注意

#### 部署命令
`wrangler pages deploy . --project-name=exam-prep-ultra`
- 首次部署自动创建 Pages 项目
- Cloudflare 会给出 `.pages.dev` 域名
- 可绑定自定义域名

#### 回滚策略
- **代码回滚**：Cloudflare Pages 支持 deployment alias，可快速回滚
- **数据库回滚**：需要 `d1 export` 备份 + 手动恢复
- **R2 回滚**：R2 有版本控制，可恢复旧版本

### 设计决策总结

| 决策 | 选择 | 原因 |
|------|------|------|
| 种子数据方式 | SQL 文件 + 手动替换哈希 | 简单可控，不依赖额外工具 |
| 幂等插入 | INSERT OR IGNORE | 可重复执行，不会覆盖已有数据 |
| 密码哈希生成 | 独立 Node.js 脚本 | 复用 register.js 的 PBKDF2 逻辑，保证一致性 |

| R2 路径命名 | banks/default.json | 语义清晰，与用户题库 banks/{userId}/{bankId}.json 一致 |
| 本地测试 | wrangler pages dev | 一条命令启动完整本地环境 |
| 部署方式 | wrangler pages deploy | Cloudflare Pages 标准部署 |

## 实际部署记录

### R2 → KV 迁移

**问题**：部署时执行 `wrangler r2 bucket create exam-questions` 报错 `Please enable R2 through the Cloudflare Dashboard`。R2 需要在 Dashboard 绑定支付信息（绑卡）才能激活，用户无法/不愿意绑卡。

**解决方案**：改用 KV 存储题目 JSON。KV 单值最大 25MB，default.json 才 700KB，完全够用。

**修改内容**：

| 文件 | 变更 |
|------|------|
| `wrangler.toml` | 删除 `[[r2_buckets]]`，新增 `[[kv_namespaces]]` binding=QUESTIONS |
| `schema.sql` | `r2_key` → `kv_key` |
| `seed.sql` | `r2_key` → `kv_key`，注释从"R2"改为"KV" |
| `functions/api/banks/index.js` | `r2Key` → `kvKey`，`env.QUESTIONS.put()`（KV 和 R2 的 put 签名相同） |
| `functions/api/banks/[id].js` | `bank.r2_key` → `bank.kv_key`，`env.QUESTIONS.delete(bank.kv_key)` |
| `functions/api/banks/[id]/questions.js` | `bank.r2_key` → `bank.kv_key`，读取逻辑从 `obj.text()` 改为直接 `await env.QUESTIONS.get()` |
| `functions/api/admin/users.js` | `SELECT r2_key` → `SELECT kv_key`，`b.r2_key` → `b.kv_key` |

**KV vs R2 API 差异**：
- R2 `get()` 返回 R2ObjectBody，需要 `await obj.text()` 转字符串
- KV `get()` 直接返回字符串，更简单
- R2 `put()` 和 KV `put()` 签名相同
- R2 `delete()` 和 KV `delete()` 签名相同

### Pages 绑定问题（关键踩坑）

**问题**：部署后访问 API 报 `Cannot read properties of undefined (reading 'prepare')`，即 `env.DB` 为 undefined。

**根因**：`wrangler pages deploy` 每次部署会用 wrangler.toml 的配置覆盖 Pages 项目设置。wrangler.toml 只在顶层声明了 `d1_databases` 和 `kv_namespaces`，但 Pages 生产环境需要这些绑定在 `env.production` 下声明。

**解决**：在 wrangler.toml 中同时声明顶层和 `env.production` 的绑定：
```toml
# 顶层（用于 wrangler pages dev 本地开发）
[[d1_databases]]
binding = "DB"
database_name = "exam-prep-db"
database_id = "e7c2a462-..."

[[kv_namespaces]]
binding = "SESSION"
id = "95a216..."

[[kv_namespaces]]
binding = "QUESTIONS"
id = "966124..."

# env.production（用于 wrangler pages deploy 生产部署）
[[env.production.d1_databases]]
binding = "DB"
database_name = "exam-prep-db"
database_id = "e7c2a462-..."

[[env.production.kv_namespaces]]
binding = "SESSION"
id = "95a216..."

[[env.production.kv_namespaces]]
binding = "QUESTIONS"
id = "966124..."
```

**验证方法**：部署后用 Cloudflare API 检查绑定是否保留：
```bash
curl -s "https://api.cloudflare.com/client/v4/accounts/{ACCOUNT_ID}/pages/projects/exam-prep-ultra" \
  -H "Authorization: Bearer {TOKEN}" | python3 -c "
import sys,json
d=json.load(sys.stdin)
c=d['result']['deployment_configs']['production']
print('D1:', json.dumps(c.get('d1_databases',{})))
print('KV:', json.dumps(c.get('kv_namespaces',{})))"
```
